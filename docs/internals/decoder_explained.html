The AES-XTS Decoder

A from-scratch tour of how this repository tries to recover plaintext from AES-XTS ciphertext by turning decryption into a constrained optimization problem and attacking it with replica-exchange (parallel tempering) Monte Carlo under an augmented-Lagrangian energy.

Honest framing. This is a research / experimental sampler. AES-256 is not broken; with full 14 rounds and unknown keys the search space is astronomically large and the engine will essentially never converge. The value of the codebase is the machinery: a circuit IR for AES-XTS, a per-byte "lowered" relaxation, and a heavily instrumented parallel-tempering engine with method-of-multipliers constraint handling, feedback-optimized temperature ladders, and MCMC convergence diagnostics. This page explains that machinery in detail.

1. What this repository actually does

Given a block of AES-XTS ciphertext (and the public sector/block indices), the decoder searches for an assignment of the unknown variables

\[ x \;=\; \big(\,\underbrace{P}_{\text{plaintext bytes}},\; \underbrace{K_1}_{\text{data key}},\; \underbrace{K_2}_{\text{tweak key}},\; \underbrace{W}_{\text{internal AES wires}}\,\big) \]

such that two things hold simultaneously:

  1. Cryptographic consistency — encrypting \(P\) under \((K_1,K_2)\) reproduces the observed ciphertext exactly; and (in the lowered model) every intermediate AES byte is internally consistent.
  2. A language prior — every plaintext byte is text ASCII (printable, or tab/newline/CR). Without this prior the problem is hopeless: for any key there exists some plaintext, so the ASCII constraint is what makes a recovered message meaningful and the search well-posed.

The whole thing is framed as energy minimization / Boltzmann sampling: define an energy \(E(x)\) that is zero exactly when \(x\) is a valid, ASCII, ciphertext-matching assignment, and use a stochastic sampler to find the low-energy (ideally zero-energy) states. Each zero-energy state is a candidate decryption that is "harvested."

Three implementations live side by side. A pure-Python reference engine (backends/python.py) defines the ground-truth semantics; a high-performance C++/OpenMP parallel-tempering engine (backends/native/solvers/parallel_tempering.cpp) does the real work; and a sibling continuous-relaxed engine (backends/native/solvers/continuous_relaxed.cpp, selected as native-continuous-relaxed / NativeContinuousRelaxedEngine) swaps the discrete residuals for a smooth relaxation and adds exact one-round Gibbs moves, approximate multi-round profile moves, and schedule-aware key proposals. All three are exposed to Python through a Cython bridge (_bridge.pyx). The Python path is the correctness oracle; the native paths are the optimizers. §14–§17 dissect the continuous-relaxed engine, which is the focus of the most recent work.

2. AES-XTS in one screen

XTS (XEX-based tweaked codebook with ciphertext stealing) is the disk-encryption mode. It uses two independent AES-256 keys: \(K_1\) encrypts data, \(K_2\) derives a per-block tweak. For the block at sector number \(s\) and block index \(j\) within that sector, encryption is:

\[ T_0 \;=\; \mathrm{AES}_{K_2}\!\big(\,\mathrm{LE}_{128}(s)\,\big), \qquad T_j \;=\; T_0 \otimes \alpha^{\,j} \ \ \text{in } \mathrm{GF}(2^{128}) \]
\[ C \;=\; \mathrm{AES}_{K_1}\!\big(P \oplus T_j\big)\;\oplus\; T_j \]

Here \(\mathrm{LE}_{128}(s)\) is the sector number as a little-endian 128-bit block (see sector_tweak_input), \(\alpha\) is the primitive element \(x\), and \(\otimes\alpha\) is the "multiply by x" operation in \(\mathrm{GF}(2^{128})\) with reduction polynomial \(x^{128}+x^7+x^2+x+1\) (the magic constant 0x87):

XtsMulX(tweak):                  // parallel_tempering.cpp
    carry = 0
    for i in 0..15:
        next_carry = (tweak[i] & 0x80) ? 1 : 0
        out[i] = (tweak[i] << 1) | carry
        carry  = next_carry
    if carry: out[0] ^= 0x87        // reduction
    return out

The data path is AES-256 itself: an initial AddRoundKey, then \(R-1\) full rounds of SubBytes → ShiftRows → MixColumns → AddRoundKey, then a final round without MixColumns. The repo carries a full, dependency-free reference of all of this (S-box, key schedule with RotWord/SubWord/Rcon, \(\mathrm{GF}(2^8)\) MixColumns) in both Python (aes_primitives.py) and C++.

Reduced rounds for positive controls. The config carries aes_rounds (1–14). At fewer rounds the cipher is weaker, so the engine can actually solve it — useful as a positive control. Because the fixture ciphertext is produced with full 14 rounds, block_specs_for_model recomputes the correct reduced-round target ciphertext from the known plaintext/key so the reduced model is validated against a matching target.

3. Why this is hard: the inverse problem

Forward encryption is cheap. The decoder is solving the inverse: it knows \(C\), the sector \(s\), and the block index \(j\), and wants \(P\) (and implicitly the keys). If the keys were known, decryption would be a one-liner — and in fact the engine exploits exactly that with its repair move. The difficulty is that \(K_1,K_2\) are 256 bits each, so the joint key space is \(2^{512}\), and AES's confusion/diffusion mean the residual landscape is extremely rugged: tiny changes to a key byte scramble every output byte.

The strategy is to convert "find a satisfying assignment" into "minimize a non-negative energy whose global minima are the satisfying assignments," then throw a good sampler at it. The ASCII prior shrinks the effective target set, and parallel tempering plus the constraint-repair move are the tools for crossing the rugged landscape.

4. The circuit intermediate representation

Everything is compiled into a compact, array-friendly IR in circuit.py. Three record types matter:

Values

Integer-ID'd quantities, each with a bit width. Inputs (plaintext/key bytes), constants (the target ciphertext), and internal "wires" are all values.

Ops

A small instruction set that computes a value from inputs: INPUT, CONST, XOR8/128, SBOX8, MIX_COLUMN_BYTE, AES256_ROUND_KEY_BYTE, XTS_MUL_X_BYTE, EXTRACT_BYTE, AES_XTS_BLOCK.

Constraints

Predicates that should hold: EQ8, EQ128, ASCII_PRINTABLE, DEFINE8. These produce the residuals the optimizer minimizes.

An Assignment holds the mutable search state as flat byte buffers: plaintext, key1, key2, and optional wires (the lowered internal values). The native engine mirrors this as AssignmentState.

The IR is deliberately flat so it serializes cleanly to the C++ side. op_table() / constraint_table() emit fixed-width integer rows; the Cython bridge then flattens ops into parallel arrays (opcodes, outputs, input_offsets, inputs, immediates, …) plus the constant pool and per-value bit widths. The C++ constructor rebuilds three crucial adjacency structures from those arrays:

  • ops_by_input_[v] — which ops consume value v (forward dataflow graph, used for delta evaluation).
  • constraints_by_value_[v] — which constraints are affected by v (including one hop through producer inputs).
  • definitions_by_right_[v] — the DEFINE8 wire definitions whose right-hand side is v (used to propagate derived wire values).

5. Two encodings: opaque vs lowered

The same XTS block can be modeled at two granularities. This choice is the single biggest structural decision in the system.

5.1 Opaque encoding

One monolithic op, AES_XTS_BLOCK, computes the predicted ciphertext for a block by calling the reference cipher directly. The only constraints are:

  • GOAL EQ128: predicted ciphertext \(=\) target ciphertext (128 bits at once).
  • ASCII per plaintext byte.

There are no internal wires. This is compact, but the energy landscape is a black box: the only moves are on plaintext/key bytes, and a single key bit flip changes all 16 ciphertext bytes unpredictably. Good for scoring, bad for local search.

5.2 Lowered encoding

Here every AES micro-step becomes its own op writing an internal wire value, and each wire carries a DEFINE8 "consistency" constraint asserting wire == recomputed value. LoweredAesEncoding.encrypt_block emits, per round: SBOX8 per byte, ShiftRows as pure wire re-routing (DEFINE8 copies), MIX_COLUMN_BYTE per output byte (a GF(2⁸) linear combination with the MixColumns coefficient row), and XOR8 AddRoundKey. The key schedule itself is exposed via AES256_ROUND_KEY_BYTE ops, also pinned to wires. The XTS tweak chain is built with XTS_MUL_X_BYTE ops, the pre-whitening and post-whitening are XOR8 wires, and each ciphertext byte gets an EQ8 constraint against the target.

The payoff: the search now has thousands of local degrees of freedom (the wires) and the optimizer can flip one wire bit, re-score only the handful of constraints it touches, and walk the landscape incrementally. The cost: many more variables and constraints, and the burden of keeping all those wires mutually consistent — which is exactly what the CONSISTENCY class measures.

The lowered builder also calls register_xts_block(...), recording each block's sector number, block index, plaintext offset, and target ciphertext. This metadata is what enables the native engine's exact constraint-repair move.

6. Constraints and their three classes

Every constraint is tagged with a class (classify_constraints), and the class controls its initial weight, its penalty coefficient, and how its multiplier is updated. There are exactly three:

ClassConstraint kindsMeaningInitial multiplier \(\lambda_i^{(0)}\)
ASCII (1)ASCII_PRINTABLEplaintext byte is text ASCIIascii_weight
CONSISTENCY (2)DEFINE8an internal wire equals the op that defines itconsistency_weight / 8
GOAL (3)EQ8, EQ128predicted ciphertext equals targetgoal_weight / 8

The division by 8 reflects that consistency/goal residuals are measured in bits (up to 8 per byte), whereas ASCII is a single 0/1 indicator, so the per-bit weighting is normalized to a comparable per-byte scale.

The class split is what allows the optimizer to treat the problem as a true constrained program: the GOAL term is the thing we want to drive to zero, while ASCII and CONSISTENCY are constraints whose multipliers are adapted (see §12). A common schedule is to start goal_weight small (a "goal ramp," default 0.1) and let the dual updates grow the constraint pressure.

7. Residuals: how violation is measured

A residual \(r_i(x) \ge 0\) measures how badly constraint \(i\) is violated; \(r_i = 0\) means satisfied. The native scorer (ResidualForConstraint) defines:

\[ r_i = \begin{cases} [\,b \notin \text{TextASCII}\,] & \text{ASCII (a single 0/1 indicator)} \\[1ex] \operatorname{popcount}(a \oplus b) & \text{EQ8 / DEFINE8 (bit-level Hamming distance)} \\[1ex] \displaystyle\sum_{k=0}^{15}\operatorname{popcount}(a_k \oplus b_k) & \text{EQ128 (Hamming over 16 bytes)} \end{cases} \]

Using Hamming distance in bits rather than a 0/1 equality is essential: it makes equality constraints smooth in the sense that getting "closer" to the target (fewer wrong bits) lowers the residual, giving the sampler a usable gradient signal even when the byte is not yet exactly right. The text-ASCII test accepts 0x09, 0x0A, 0x0D and the printable range 0x20–0x7E.

The aggregate score for an assignment is

\[ H(x) \;=\; \sum_i r_i(x), \qquad \text{violations} = \#\{ i : r_i(x) > 0\}. \]

An assignment is feasible iff \(H(x) = 0\). Note \(H\) is the raw, unweighted Hamming score used for feasibility tests and diagnostics; the optimizer instead minimizes the weighted energy below.

The relaxations module

The Python side abstracts residual computation behind a Relaxation protocol. DiscreteLocalRelaxation is the default and matches the native scorer exactly (integer bit counts). ByteDistributionRelaxation is an experimental continuous relaxation that lifts each byte to a probability vector over its 256 possible values, pushes distributions through the S-box (\(\mathbb{E}[\,\mathrm{SBOX}\,]\) as a permutation pushforward), and exposes gradients of equality factors in distribution space — scaffolding for a future differentiable/Langevin engine rather than something the PT engine uses today.

The native continuous-relaxed engine takes a lighter-weight route to a smooth landscape: rather than full 256-way distributions it keeps each residual a single scalar but makes it graded — normalizing every Hamming distance to \([0,1]\) and, crucially, replacing the flat 0/1 ASCII test with a graded bit-distance to the nearest printable byte. That graded surrogate is the relaxation the production engine actually optimizes; it is developed in §14.

8. The augmented-Lagrangian energy

This is the mathematical heart of the optimizer. The engine does not minimize the raw Hamming score; it minimizes an augmented Lagrangian (a method-of-multipliers objective) over the residuals (Energy in the C++):

\[ \boxed{\;E(x) \;=\; \sum_i \Big[\, \lambda_i\, r_i(x) \;+\; \tfrac{1}{2}\,\rho_i\, r_i(x)^2 \,\Big]\;} \]

Each constraint contributes two terms:

  • a linear multiplier term \(\lambda_i r_i\) — the Lagrange multiplier estimate, adapted over time (the multipliers_ array);
  • a quadratic penalty term \(\tfrac12\rho_i r_i^2\) — with penalty coefficient \(\rho_i\) (the penalties_ array, set to mu in fixed mode or to a per-class \(\rho\) in scheduled mode).

This is precisely the classic augmented Lagrangian for the equality-constrained program "minimize 0 subject to \(r_i(x)=0\ \forall i\)." Why both terms? The pure quadratic penalty method needs \(\rho\to\infty\) to enforce constraints exactly (ill-conditioned), while the multiplier term lets the constraints be satisfied with finite penalty by driving \(\lambda_i\) toward the true Lagrange multipliers. The combination is well-conditioned and is the standard method of multipliers (Hestenes–Powell).

Because every \(r_i\ge 0\) and the global target is \(r_i=0\), the energy is non-negative and \(E(x)=0 \iff H(x)=0 \iff x\) is feasible. So minimizing \(E\) and hunting for feasible assignments are the same objective, just reweighted to shape the landscape.

The Boltzmann distribution this energy induces at inverse temperature \(\beta\) is \(\pi_\beta(x) \propto e^{-\beta E(x)}\). As \(\beta\to\infty\) the distribution concentrates on the global minima (feasible assignments); at small \(\beta\) it is broad and easy to explore. Parallel tempering is the device that gets the best of both.

9. Parallel tempering (replica exchange)

The engine runs \(N\) replicas of the system simultaneously, each held at a different temperature on a ladder. Hot replicas roam freely across energy barriers; cold replicas refine deep minima. Periodically neighbors swap configurations, ferrying good states discovered by hot replicas down to cold ones and vice versa. This is the textbook cure for rugged, multimodal landscapes.

9.1 The temperature ladder

Temperatures are geometric between t_min and t_max (TemperatureLadder):

\[ T_r \;=\; T_{\min}\,\Big(\tfrac{T_{\max}}{T_{\min}}\Big)^{\frac{r}{N-1}}, \qquad r = 0,1,\dots,N-1. \]

A geometric (constant-ratio) spacing is the standard default because, for many systems, it produces roughly uniform swap-acceptance across the ladder.

9.2 Effective inverse temperature with Lagrange scaling

A distinctive twist: each replica also carries a per-rung Lagrange scale \(\sigma_r\), geometric from lambda_scale_cold (rung 0) to lambda_scale_hot (rung \(N-1\)). The effective inverse temperature is

\[ \beta_r \;=\; \frac{\sigma_r}{T_r} \qquad(\text{\texttt{EffectiveBeta}}). \]

This lets hot replicas explore with weakened constraint pressure (when the scales differ) while cold replicas enforce constraints strongly — a way to decouple "how much do I care about constraints" from raw temperature. With the default scales \(=1\) it reduces to ordinary PT, \(\beta_r = 1/T_r\).

9.3 One sweep

An epoch is sweeps sweeps followed by a dual update. A single sweep (RunEpoch body) is:

for each sweep:
    // 1. Local moves, one per replica, in parallel (OpenMP)
    #pragma omp parallel for
    for replica in 0..N-1:
        StepReplica(replica)            // Metropolis proposal + accept/reject
    // 2. Replica exchange across the whole ladder
    AttemptSwaps()
    // 3. Bookkeeping: replica flow, energy moments
    UpdateReplicaFlow()
    RecordEnergyStats()
// after the sweeps:
DualUpdate()                            // adapt multipliers / penalties
OptimizeTemperatureLadder()             // optional feedback ladder retuning

9.4 The Metropolis step (per replica)

StepReplica proposes one candidate (via a move, §10), computes its energy, and accepts with the Metropolis rule at the replica's effective \(\beta\):

\[ P_{\text{accept}} \;=\; \min\!\Big(1,\; \exp\big[-\beta_r\,(E_{\text{cand}} - E_{\text{cur}})\big]\Big). \]

Downhill moves (\(E_{\text{cand}} \le E_{\text{cur}}\)) are always accepted; uphill moves are accepted with probability \(e^{-\beta_r \Delta E}\). This is exactly detailed balance for \(\pi_{\beta_r}\) — provided the proposal is symmetric, which the bit-flip moves are (see the caveat on the plaintext move in §10).

9.5 The swap step

AttemptSwaps walks adjacent pairs \((r, r{+}1)\) and exchanges their configurations with the replica-exchange acceptance probability. With \(\Delta\beta = \beta_r - \beta_{r+1}\) and \(\Delta E = E_r - E_{r+1}\):

\[ \log P_{\text{swap}} \;=\; (\beta_r - \beta_{r+1})\,(E_r - E_{r+1}) \;=\; \Delta\beta\,\Delta E, \qquad P_{\text{swap}} = \min(1, e^{\Delta\beta\,\Delta E}). \]

This is the standard Metropolis criterion for swapping two temperatures in the joint product distribution \(\pi_{\beta_r}\!\otimes\pi_{\beta_{r+1}}\); it preserves the stationary distribution of the whole ensemble. When a swap is accepted the engine swaps everything tied to that replica — configuration, cached score, energy, value buffer, key cache, and flow direction — so each rung remains internally consistent.

10. The moves and updates

Proposals come in two flavors: random mutations (four families) and the constraint-repair move. StepReplica first tries the repair move with probability repair_move_prob (if the lowered XTS metadata is present); otherwise it calls Mutate.

10.1 Random mutation families

Mutate rolls a uniform \(u\in[0,1)\) and picks one of the local moves, with a schedule-aware word swap mixed into the key branch:

#FamilyTriggerWhat it does
0wire bit flip\(u < 0.65\) and wires existpick a random wire byte, XOR a random single bit. The workhorse local move of the lowered model.
1plaintext ASCII resample\(0.65 \le u < 0.9\)pick a random plaintext byte, replace it with a uniformly drawn text-ASCII character.
2key bit flipkey branch, non-swappick key1 or key2 (50/50), pick a random byte, XOR a random single bit.
3key word swapkey branch, 50% when a 32-byte key existspick key1 or key2, choose two of the eight AES-256 4-byte words, and swap the words. This is symmetric but much larger than a single-bit avalanche move.

The bit flips and word swaps are symmetric proposals (applying the same mutation again returns to the old state), so the plain Metropolis ratio in §9.4 is correct. The plaintext resample is almost symmetric; the source comment flags that strict reversibility only holds once the byte is already within the ASCII support. In practice it acts as a fast "repair" toward the ASCII prior.

10.2 The constraint-repair (Gibbs-style) move

This is the cleverest move and the reason the lowered block metadata exists. Given the replica's current keys, the goal constraints for a block can be satisfied exactly and all at once by simply decrypting the target ciphertext. RepairMove picks a block and computes:

\[ T \;=\; \mathrm{AES}_{K_2}^{(R)}\!\big(\mathrm{LE}_{128}(s)\big) \otimes \alpha^{\,j}, \qquad \widehat{P} \;=\; \mathrm{AES}^{-1,(R)}_{K_1}\!\big(C \oplus T\big) \;\oplus\; T, \]

and writes \(\widehat{P}\) into that block's plaintext. By construction the predicted ciphertext now equals \(C\) for that block, so every GOAL residual there drops to zero in one shot — for any round count \(R\). This is effectively a block Gibbs update of the plaintext conditioned on the keys.

The strategic consequence is stated in the source itself: with the repair move, feasibility reduces to finding a key whose decryption is ASCII. The goal constraints become free; the search collapses onto the key space and the ASCII prior. The exact inverse cipher DecryptReducedRound (InvShiftRows / InvSubBytes / InvMixColumns with the \(\{9,11,13,14\}\) GF coefficients) is the mirror of EncryptReducedRound.

10.3 Replica initialization

At construction, replica 0 starts from the supplied initial assignment; every other replica is the initial assignment after one Mutate (and a wire re-derivation if the mutation touched a non-wire), giving a diverse starting ensemble. Each replica also gets its own RNG stream seeded by seed + φ·(i+1) with the golden-ratio constant, so the OpenMP-parallel replica steps are independent and reproducible.

11. Incremental (delta) scoring

Re-evaluating the entire lowered circuit on every proposal would be ruinously slow — there are thousands of ops. For the common case (a wire bit flip), the engine instead does incremental evaluation in ScoreWireFlipAffected, the key performance trick.

Each replica keeps a persistent ValueBuffer with the current value of every op output. When wire \(v\) changes, the engine:

  1. writes the new byte into \(v\) and seeds a work queue with it;
  2. performs a BFS over the dataflow graph ops_by_input_: for each op consuming a changed value, recompute it (EvaluateOpInto); if its output actually changed, enqueue it. A monotonically increasing mark stamp per replica prevents revisiting and avoids clearing arrays each time;
  3. collects the set of affected constraints via constraints_by_value_ and re-scores only those, updating hamming_score and violations by the residual deltas;
  4. records an undo log of \((\text{value id}, \text{old bytes})\) pairs so that if the Metropolis step rejects, RestoreValues rolls back in \(O(\text{touched})\) instead of recomputing.

So an accepted/rejected wire flip costs work proportional to the size of its dependency cone, not the whole circuit. Non-wire moves (plaintext/key changes, or the repair move) fall back to a full EvaluateValuesAndDerive + ScoreFromValues, because they can change many values at once.

Wire derivation

EvaluateValuesInternal can also derive wires: when an op output feeds a DEFINE8 whose left side is a wire, the computed value is propagated into the wire buffer (transitively through chained definitions). This is how a fresh plaintext/key assignment gets a fully consistent set of internal wires before scoring.

12. Dual updates: adapting multipliers and penalties

After each epoch, DualUpdate performs the dual ascent step of the method of multipliers. First it computes the replica-scale-weighted mean residual of each constraint across the ensemble:

\[ \bar r_i \;=\; \frac{\sum_r \sigma_r\, r_i^{(r)}}{\sum_r \sigma_r}. \]

Then it updates, depending on dual_mode:

12.1 Fixed mode

A constant-step gradient ascent on the dual, applied to ASCII and consistency constraints (the goal multiplier is steered by the goal ramp instead):

\[ \lambda_i \;\leftarrow\; \max\!\big(0,\; \lambda_i + \eta\,\bar r_i\big), \qquad \eta = \texttt{dual\_eta}. \]

This is exactly the classic multiplier update \(\lambda \leftarrow \lambda + \eta\,c(x)\): constraints that remain violated on average steadily accrue larger multipliers, increasing the energy cost of violating them, which pressures the cold replicas to satisfy them. The penalty \(\rho_i = \mu\) is constant.

12.2 Scheduled mode

A per-class adaptive scheme that decides, for each class, whether to update the multiplier or escalate the penalty based on whether the violation is shrinking fast enough. With per-class mean violation \(v_c\) and previous value \(v_c^{\text{prev}}\):

\[ \text{if } v_c \le \kappa\, v_c^{\text{prev}} \;(\text{shrinking}):\quad \lambda_i \leftarrow \max(0,\ \lambda_i + \rho_c\, \bar r_i)\ \ \forall i\in c \]
\[ \text{else (stalled)}:\quad \rho_c \leftarrow \max\!\big(\rho_c\cdot\gamma,\ \rho_c + \epsilon\big) \]

where \(\kappa=\) scheduled_violation_shrink (e.g. 0.75) and \(\gamma=\) scheduled_rho_growth (e.g. 2.0). This mirrors the standard convergence test in augmented-Lagrangian solvers: if the constraint is converging, take a multiplier step; if it has stalled, increase the penalty to squeeze harder. After 6 consecutive escalations with a class still above scheduled_eta_target, the engine raises an infeasibility_suspected flag — a principled "this window probably has no ASCII-consistent solution" signal.

Either way it finishes by refreshing the penalty cache and recomputing all replica energies (because the energy function itself just changed), then resetting the energy statistics.

13. Feedback temperature-ladder adaptation

A geometric ladder is a good default, but the optimal ladder depends on where the energy landscape's bottlenecks are. With ladder_mode = "feedback", OptimizeTemperatureLadder retunes the interior temperatures using the feedback-optimized method (Katzgraber–Trebst–Huse–Troyer), driven by replica flow.

13.1 Tracking replica flow and round trips

Each replica is labeled with a direction: it is set to "up" when it touches the bottom rung and "down" when it touches the top rung. A full bottom→top→bottom excursion is a round trip, the canonical measure of how well the ladder mixes. UpdateReplicaFlow increments per-rung up/down counters and round-trip counts every sweep.

13.2 The optimization

Let \(f_i\) be the fraction of time rung \(i\) is moving "up": \(f_i = \frac{n^\uparrow_i}{n^\uparrow_i + n^\downarrow_i}\). The method defines a target density proportional to \(\sqrt{|\,df/dT\,|}\), which places more rungs where the flow changes fastest (the bottlenecks). The code builds the cumulative measure

\[ g(i) \;=\; \sum_{k\le i} \sqrt{\,|f_k - f_{k-1}| + \varepsilon\,}, \qquad \widehat g(i) = g(i)/g(N{-}1), \]

then redistributes the interior temperatures so they are equally spaced in this normalized cumulative coordinate (interpolating the old temperatures against \(\widehat g\)), holding the endpoints fixed and keeping the ladder strictly increasing. The retuning only fires after a burn-in, on a fixed epoch interval, and only once enough round trips have accumulated to make the flow estimate meaningful; afterward it resets the flow counters.

14. The continuous relaxation: smooth residuals & energy

The continuous-relaxed engine (continuous_relaxed.cpp) is a sibling of the PT solver, not a replacement. It shares the circuit IR, the augmented-Lagrangian skeleton, the replica-exchange swaps, the harvesting logic, and the entire diagnostics layer. What it changes is the shape of the energy and the repertoire of moves. This section covers the energy; §15–§17 cover the moves and the schedule.

14.1 Graded residuals

The PT scorer of §7 returns integer Hamming bits, and its ASCII residual is a flat 0/1 indicator. The relaxed engine instead computes a smooth residual \(\tilde r_i(x)\in[0,1]\) for every constraint (SmoothResidual):

\[ \tilde r_i = \begin{cases} \dfrac{\operatorname{TextAsciiDist}(b)}{8} & \text{ASCII} \\[2.2ex] \dfrac{\operatorname{popcount}(a \oplus b)}{8} & \text{EQ8 / DEFINE8} \\[2.2ex] \dfrac{1}{8c}\displaystyle\sum_{k=0}^{c-1}\operatorname{popcount}(a_k \oplus b_k) & \text{EQ128 (over } c \text{ bytes)} \end{cases} \]

The equality cases are just the §7 Hamming distances rescaled to a per-byte \([0,1]\) range. The decisive change is the ASCII term. The hard scorer answers a flat yes/no — is this byte printable? — which gives the sampler no gradient: every non-printable byte looks equally bad. TextAsciiDistance(b) instead returns the minimum number of bit flips needed to turn \(b\) into the nearest text-ASCII byte (any of 0x09, 0x0A, 0x0D or the printable range 0x20–0x7E). A byte that is one bit away from printable now has residual \(1/8\) rather than \(1\), so "almost printable" is rewarded and the ASCII domain becomes a smooth basin instead of a cliff. This graded ASCII distance is the heart of the relaxation.

14.2 Two scores in one

Each replica carries a ContinuousRelaxedScore bundling both representations:

  • the exact, discrete ScoreData (hard) — integer residuals, hamming_score, and violations, used unchanged for feasibility tests, harvesting, and diagnostics;
  • the smooth residual vector and its running sum smooth_sum — used only to drive the optimizer's energy.

So nothing about what counts as a solution is relaxed: a state is feasible iff its hard \(H(x)=0\), exactly as before. The relaxation reshapes the slopes the sampler descends, never the location of the global minima.

14.3 The relaxed energy

Energy is the same augmented Lagrangian as §8, but evaluated on the smooth residuals:

\[ E(x) \;=\; \sum_i \Big[\, \lambda_i\, \tilde r_i(x) \;+\; \tfrac12\, \rho_i\, \tilde r_i(x)^2 \,\Big]. \]

One bookkeeping consequence of pre-normalizing the residuals: the initial multipliers are the raw class weights ascii_weight, consistency_weight, goal_weight with no division by 8. (The PT engine divides the weight by 8 because its residuals run \(0\!-\!8\); the relaxed engine divides the residual by 8 and keeps the weight — the two conventions land on the same per-byte scale.)

Because each \(\tilde r_i = 0 \iff r_i = 0\), the relaxed energy still satisfies \(E(x)=0 \iff x\) is feasible. The smooth and hard objectives share their zero set; the smooth one simply has a friendlier interior.

15. Profile-guided key-space moves

The repair move already reduced feasibility to "find a key whose decryption is ASCII." But finding that key by flipping bits is hopeless: AES's avalanche means a single key-bit change re-randomizes every plaintext byte after repair, so the post-repair ASCII residual behaves like noise and bit/byte flips merely random-walk. The relaxed engine's signature contribution is a set of moves that resample or profile larger chunks of key space. The one-round moves are exact Gibbs conditionals; for multi-round XTS windows, the engine now uses a cached profile-energy approximation over \(K_1\) to make \(K_2\) moves informative instead of nearly flat.

15.1 All-block ADMM repair, cached

ApplyAdmmRepair generalizes the single-block repair move to the whole window at once: for every registered XTS block it derives the tweak from \(K_2\), pre-whitens the target ciphertext, decrypts under \(K_1\), and post-whitens, writing the exact plaintext for that block. The expanded key schedules and the result are memoized in a per-replica KeyCache keyed on \((K_1,K_2)\); RepairReplicaIfNeeded re-runs it only when the key actually changed, so unchanged replicas pay nothing.

The fast ASCII energy. After repair, every goal and consistency residual is identically zero, so the entire Lagrangian collapses to the ASCII-printability term. AsciiEnergyFromPlaintext reads that energy straight off the plaintext bytes without re-deriving a single wire, making the 256-way and \(256^2\)-way key scans below roughly an order of magnitude cheaper than a full circuit evaluation.

15.2 The one-round decryption identity

For one round, each plaintext byte is an explicit closed form in just two key1 bytes. With \(s=\operatorname{InvShiftRows}(\text{dst})\):

\[ P_b[\text{dst}] \;=\; \operatorname{InvSbox}\!\big[\, C_b[s] \,\oplus\, T_b[s] \,\oplus\, rk_1[s] \,\big] \;\oplus\; rk_0[\text{dst}] \;\oplus\; T_b[\text{dst}], \]

where (for AES-256 the first two round keys are the raw key) \(rk_0[\text{dst}]=K_1[\text{dst}]\) and \(rk_1[s]=K_1[16+s]\), and \(T_b\) is block \(b\)'s tweak. The two key bytes \(rk_0[\text{dst}]\) and \(rk_1[s]\) are coupled through the S-box, which is exactly why no single-byte move can place them jointly — and why the engine moves them as a pair.

15.3 Move 1 — the key-column pair block move

KeyColumnPairBlockStep picks a destination position dst and considers the two key1 bytes \((rk_0[\text{dst}], rk_1[s])\) that determine plaintext byte dst in every block of the window (all blocks share the key). It scans all \(256\times256\) value pairs, accumulating the summed ASCII energy across the window, then resamples the pair from the exact Boltzmann conditional at the replica's \(\beta_r\):

\[ \Pr(rk_0,\,rk_1) \;\propto\; \exp\!\Big[-\beta_r \sum_b \big(\lambda_b \tilde r_b + \tfrac12\rho_b \tilde r_b^2\big)\Big]. \]

With probability CR_BLOCK_GREEDY it instead takes the exact \(\arg\min\) pair (a "greedy polish"), because pure Gibbs concentration is capped by the many near-printable-but-infeasible pairs and would otherwise leave a residual ASCII plateau. Since the 16 positions use disjoint key1 bytes, greedily fixing each is the exact per-position argmin and drives the entire key1-given-key2 sub-problem to zero. This is the move that actually descends to a feasible one-round key.

15.4 Move 2 — the key2 tweak-refit on a profile energy

Moving \(K_2\) is subtler: \(K_2\) sets the tweaks, and a bare \(K_2\)-byte change is scored against a \(K_1\) that was fitted to the old tweaks, so every proposal looks catastrophic and is rejected — the chain can never explore \(K_2\). The fix is to score \(K_2\) by its profile energy, the best it could do if \(K_1\) were re-optimized:

\[ E^\star(K_2) \;=\; \min_{K_1} E(K_1, K_2). \]

GreedyKey1ForKey2 computes \(E^\star\) exactly (per-position argmin over \(K_1\) for the proposed tweaks — exact because the positions are independent in \(K_1\)). KeyTweakRefitStep then perturbs 1–2 bytes of \(K_2\), recomputes \(E^\star\), and accepts on \(\Delta E^\star\) with the Metropolis rule at \(\beta_r\). This is what lets the sampler discover a \(K_2\) whose tweaks admit an all-printable decryption — the prerequisite for harvesting a feasible key. Because a refit costs ~16× a block move, it runs only on the cold replicas (temperature \(\le\) t_min · CR_REFIT_COLD_RATIO), where feasible solutions are actually harvested.

15.5 Move 3 — multi-round profile Gibbs

For \(R\ge2\), MixColumns couples the four bytes of a column, so the closed-form one-round identity above is gone and byte-wise Gibbs on a single key byte is often too flat across a 10-block window. The multi-round path keeps the same profile energy idea but computes it approximately: KeyProfileCache stores a \(32\times256\) table of repaired ASCII energies for candidate \(K_1\) bytes, ProfileKey1ForKey2MultiRound performs coordinate-descent sweeps over those rows, and KeyTweakProfileGibbsStep scans 256 values of one \(K_2\) byte using the profiled \(K_1\) for each value. The table is conservatively invalidated when the source key changes; this favors correctness first, while leaving room for narrower key-schedule dependency refreshes later.

Key1ProfileGibbsStep is the cheaper companion move: it refreshes one \(K_1\) profile row, samples the selected key byte from the cached 256-way Boltzmann conditional, then repairs and fully rescores the replica. The multi-round profile path is selected only for XTS windows with both AES-256 keys present; non-XTS circuits still fall back to the generic byte-wise move.

15.6 Move 4 — byte-wise key Gibbs (the fallback)

KeyByteGibbsStep remains the general-purpose fallback. It picks one key byte (in \(K_1\) or \(K_2\)), scores all 256 candidate values — via the fast ASCII energy when repair is active, or a full re-derivation otherwise — and resamples it from the exact conditional at \(\beta_r\). It is still useful for non-XTS circuits and as a fallback when the specialized profile moves cannot run.

15.7 The move mix

When a step elects a key move (probability key_gibbs_prob), the one-round branch chooses among the exact/refit moves; the split is env-tunable for A/B sweeps:

Roll \(u\)MoveKnobEffect
\(u < \) CR_REFIT_PROBtweak refitdefault 0.0search \(K_2\) with \(K_1\) re-optimized (cold replicas only)
next CR_BLOCK_PROBcolumn-pair blockdefault 0.6exact joint Gibbs on the S-box-coupled \(K_1\) pair
next CR_KEY2BLOCK_PROBkey2 column-pair blockdefault 0.0one-round mirror move on the tweak key \(K_2\)

For \(R\ge2\), the XTS branch instead chooses between KeyTweakProfileGibbsStep and Key1ProfileGibbsStep; CR_PROFILE_K2_PROB (default 0.5) controls how often the more expensive \(K_2\)-profile scan is used, and CR_PROFILE_K1_SWEEPS controls the number of coordinate-descent sweeps when approximating \(E^\star(K_2)\).

Setting key_gibbs_prob = 0 disables the specialized key Gibbs/profile branch and leaves exploration to the gradient-biased proposal pool. The conditionals are most identifiable when the window spans several blocks across multiple sectors, so the key bytes are over-determined; CR_FREEZE_KEY2 pins \(K_2\) for ablations.
Two-round smoke result. On a realistic 10-block fixture window rebuilt with aes_rounds=2, a bounded inversion smoke from a non-true key reduced the hard residual from 653 to 76 while exercising key_tweak_profile_gibbs, key_word_swap, and key_schedule_chain. It did not guarantee full recovery in that budget, but it verified the end-to-end inversion path, cache consistency, and move instrumentation.

16. Gradient-biased ensemble proposals

When a step does not take a key Gibbs move, the relaxed engine does not propose a single mutation and accept/reject it. Instead it builds a small pool of candidates, weights them by a gradient-biased score, samples one, and then applies an ordinary Metropolis test to it (BuildActionWeightsSampled → StepReplica).

16.1 The candidate pool

Up to kSampledCandidatePool = 8 distinct actions are drawn from up to kSampledCandidateAttempts = 64 samples (deduplicated by an ActionKey hash). Each candidate is scored to energy \(E_{\text{cand}}\) and assigned a weight

\[ w \;=\; \exp\!\Big[\, \tfrac12\big(E_{\text{base}} - E_{\text{cand}}\big) \;-\; \tfrac14\, \text{sboxBias} \,\Big]. \]

One action is then selected with probability \(\propto w\), and accepted with the standard rule \(P=\min(1, e^{-\beta_r \Delta E})\). The pool therefore biases proposals toward energy-lowering, S-box-aware moves (a "gradient-biased ensemble" step) while the Metropolis correction keeps the accept/reject stochastic.

16.2 Seven action families

SampleRandomAction and the key-profile branch report seven move families (the family index also keys the per-family acceptance stats):

#FamilyWhat it flips
0wire bit flipa random non-linear-determined wire byte (see §16.3)
1plaintext bit flipone bit of a random plaintext byte
2key bit flipone bit of a random \(K_1\)/\(K_2\) byte
3S-box-input wire flipa wire that feeds an SBOX8 op — gets the S-box bias below
4key chain flipthe same bit across key bytes spaced 8 apart — targets the AES-256 key-schedule word structure
5key word swapswap two full 4-byte AES-256 words in \(K_1\) or \(K_2\), then repair and fully rescore
6key profile Gibbsmulti-round \(K_1\)/\(K_2\) profile Gibbs moves selected by key_gibbs_prob

16.3 S-box table bias and pinned wires

Two precomputations sharpen the wire moves:

  • S-box flip table (SboxTableBias): a \(256\times256\times8\) table sbox_flip_delta_[in][target][bit] stores the exact change in the S-box output's Hamming distance to its target when bit bit of the input is flipped. For a family-3 proposal the engine sums this delta over the affected S-box constraints and folds it into the proposal weight (the \(-\tfrac14\,\text{sboxBias}\) term), steering S-box-input flips toward the S-box truth instead of away from it.
  • Linear-determined wires (linear_determined_wire_): wires that are outputs of linear ops (XOR, MIX_COLUMN_BYTE, XTS_MUL_X_BYTE, EXTRACT_BYTE, …) are pinned by linear propagation and would be overwritten on the next derive, so flipping them is wasted work. The wire family skips them and spends its entropy on the genuinely free, S-box-coupled wires.

16.4 Incremental scoring of a pooled flip

Scoring eight candidates per step would be ruinous without the delta machinery of §11. TryScoreActionDelta gates the fast path to single S-box/wire flips; ScoreWireFlipAffected then BFS-walks only the dependency cone and updates both the smooth residuals (and smooth_sum) and the hard ScoreData incrementally, using per-replica mark stamps and an undo log. Plaintext/key/chain/word-swap moves and the repair fall back to a full EvaluateValuesAndDerive + ScoreRelaxed.

17. Adam dual ascent & population annealing

17.1 Adam on the multipliers

The relaxed DualUpdate keeps the per-class penalty escalation of the scheduled mode (§12.2) — if a class's mean violation stalls instead of shrinking, its \(\rho_c\) is multiplied by scheduled_rho_growth — but it replaces the plain constant-step multiplier ascent with an Adam optimizer on the dual. For each ASCII/consistency constraint it tracks first- and second-moment estimates of the mean residual \(\bar r_i\), bias-corrects them, and adds a heavy-ball momentum term:

\[ m_i \leftarrow \beta_1 m_i + (1-\beta_1)\,\bar r_i, \qquad v_i \leftarrow \beta_2 v_i + (1-\beta_2)\,\bar r_i^{\,2}, \]
\[ \lambda_i \;\leftarrow\; \max\!\Big(0,\; \lambda_i \;+\; \eta\,\frac{\hat m_i}{\sqrt{\hat v_i}+\varepsilon} \;+\; 0.9\,(\lambda_i - \lambda_i^{\text{prev}})\Big), \]

with \(\beta_1=0.9,\ \beta_2=0.999\) and \(\eta=\) dual_eta. The per-coordinate adaptive step makes the multiplier schedule far less sensitive to the raw residual scale than the fixed-step ascent of the PT engine.

17.2 Population annealing

Where the PT engine retunes its ladder by feedback (§13), the relaxed engine keeps a fixed geometric ladder and instead resamples the ensemble. After each epoch's dual update, PopulationAnneal draws a fresh population of replicas with replacement, each picked with probability proportional to its cold-temperature Boltzmann weight:

\[ \Pr(\text{clone replica } i) \;\propto\; \exp\!\big[-\beta_0\,(E_i - E_{\min})\big]. \]

This sequential-Monte-Carlo step culls high-energy replicas and clones promising ones, concentrating compute on the basins that look most feasible — a natural fit for an engine whose whole job is to find the rare ASCII-consistent key.

Marginals require a real solution. The relaxed Metrics() adds one more gate to marginal_trusted: at least one exact feasible solution must have been harvested (total_harvested_ > 0). ESS and R-hat only certify that the cold chain moved, not that it moved toward the truth — a replica can be sharply but wrongly confident while never visiting a feasible key — so key-information is reported as trustworthy only once a zero-residual key has actually been seen.

18. Diagnostics: the instrumentation layer

A large fraction of the engine exists to measure the search. Metrics() assembles a rich diagnostics struct (surfaced to Python as a dict). The most mathematically interesting pieces:

18.1 Thermodynamic integration → relative \(\log Z\)

Per rung the engine keeps a running mean and variance of energy via Welford's algorithm (RecordEnergyStats). Treating the rungs as samples of \(\langle E\rangle_\beta\) along the inverse-temperature axis, it estimates the (relative) log partition function by the thermodynamic-integration identity \(\frac{d\log Z}{d\beta} = -\langle E\rangle_\beta\), integrated by the trapezoid rule (LogZEstimate):

\[ \log Z(\beta) \;\approx\; \log Z_0 \;-\; \int_0^{\beta} \langle E\rangle_{\beta'}\, d\beta', \qquad \log Z_0 = (\text{state bits})\cdot \ln 2. \]

The reference \(\log Z_0\) is the log count of all plaintext+key states (the \(\beta=0\) uniform distribution). The result is explicitly documented as a relative thermodynamic diagnostic over plaintext+key bits, not an absolute feasible-solution count.

18.2 Key marginals and information content

On the cold chain (rung 0), after a burn-in, RecordAcceptedKeyStats maintains a sliding window of accepted full keys and counts, per key bit, how often it is 1. Let \(\hat p_b\) be the Laplace-smoothed marginal of bit \(b\) (smoothing \(\alpha=\) marginal_alpha over a window of marginal_window visits). From these it computes:

  • max deviation \(\max_b |\hat p_b - \tfrac12|\) — how far any key bit has drifted from uniform;
  • key information \(\sum_b \big(1 - H_2(\hat p_b)\big)\) bits, where \(H_2\) is the binary entropy — total bits of "knowledge" the sampler has accumulated about the key;
  • a null baseline: the expected information a uniform key would spuriously show under finite sampling, computed exactly from the binomial distribution; the reported information subtracts this baseline so noise doesn't masquerade as signal.

18.3 Are the marginals trustworthy? ESS and R-hat

To avoid over-interpreting an unconverged chain, the engine computes two standard MCMC diagnostics on the cold-key Hamming-weight trace:

  • Effective sample size via the integrated autocorrelation time \(\tau\): \(\mathrm{ESS} = n / \big(2\tau\big)\), with \(\tau = 1 + 2\sum_{\ell\ge1}\rho_\ell\) summed until the autocorrelation goes non-positive (or the \(6\tau\) self-consistent window is reached).
  • Gelman–Rubin \(\hat R\) by splitting the trace in half and comparing between-chain and within-chain variance: \(\hat R = \sqrt{\hat V / W}\).

The marginals are flagged trusted only if there are enough distinct cold keys, \(\mathrm{ESS} \ge 25\), and \(\hat R \le 1.1\) — the usual convergence thresholds. Otherwise the reported key-information is forced to zero.

18.4 Structural and acceptance diagnostics

ComputeAlgebraCounts tallies linear vs nonlinear ops (XOR, MixColumn, XTS-mul, extract are \(\mathrm{GF}(2)\)-linear; only the S-box is nonlinear), variable bit counts, and factor counts — a cheap structural readout of how "algebraic" the instance is. The engine also tracks per-family proposal attempts/accepts, per-pair swap attempts/accepts, per-rung round trips, and (when profile is on) nanosecond timers for every phase of the loop.

19. Harvesting feasible solutions

Whenever an accepted move drives a replica to hamming_score == 0, MaybeHarvest fires (under an OpenMP critical section). It keys the assignment by its concatenated \((K_1,K_2)\) bytes and inserts it into the feasible set only if that key string is new (seen_keys_ deduplication). DrainFeasible later hands these out to the caller. The CLI (sample_blocks.py) loops epochs, draining feasible samples and writing each as a JSON record (keys hex, plaintext hex, per-block plaintext) until a target count is reached, dumping the full metrics struct after every epoch.

20. The end-to-end pipeline

  1. Fixture: a known plaintext (e.g. lorem ipsum) is AES-XTS encrypted to produce ciphertext + metadata JSON (sector size, start sector, key hex).
  2. Load & window: load_fixture reads it; block_specs_for_fixture slices a window of blocks, computing each block's sector number and in-sector index.
  3. Build circuit: build_xts_block_collection emits the opaque or lowered circuit with shared key1/key2 buffers across all blocks plus ASCII constraints on plaintext.
  4. Initialize: random ASCII plaintext + random keys (or, for a sanity-check positive control, a fixture warm start that intentionally leaks the key).
  5. Search: NativeParallelTemperingEngine wraps the C++ engine; the driver runs epochs of sweeps, each epoch ending in a dual update and optional ladder retune.
  6. Harvest: feasible assignments are drained and written out; metrics are logged per epoch.
Correctness oracle. The native scorer is continuously cross-checked: NativeParallelTemperingEngine.evaluate compares the C++ residuals to the Python circuit.evaluate residuals and raises if they ever disagree; with validate_python_epoch the per-epoch score is checked too. The pure-Python engine is the semantic ground truth, the C++ engine is the fast path.
